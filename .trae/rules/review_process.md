## 设计文档评审流程

1. 作者完成设计文档
2. 提交到项目进行交叉审查
3. 对照 `.trae/specs/foundry-v1/checklist.md` 逐项验证
4. 所有 🔴 项通过后方可进入实现阶段
5. 评审意见必须记录在文档的修订历史中
6. 如评审意见存在分歧，由项目架构师最终裁决；裁决结果必须记录在评审意见中并说明理由

## 评审结果分级

评审发现的问题分为两级：

| 级别 | 标记 | 含义 | 处理要求 |
|------|------|------|---------|
| 阻塞项 | 🔴 B-N | 必须修复，否则文档不可通过评审 | 修复后必须重新验证 |
| 建议项 | 🟡 S-N | 建议修复，但不阻塞评审通过 | 可选择修复或记录为待决问题 |

**编号规则**：B/S-N，N 为序号，如 B-1、S-3。同一文档内编号连续。

## 评审维度

### 工程哲学符合性

对照 `engineering_checklist.md` 逐项验证：

| 维度 | 验证项 |
|------|--------|
| Flow First | 行为是否附着在流程中？Agent 是否拥有流程控制权？流程是否能独立运行？ |
| Deterministic Over Smart | 相同输入输出是否确定？是否依赖模型聪明程度？失败模式是否明确？ |
| Artifact Over Conversation | 输出是否结构化？是否定义了类型和 Schema？是否存在无结构文本？ |
| Agent Is Replaceable | 是否可通过配置替换？是否依赖特定模型特性？是否支持动态禁用？ |

### 跨文档一致性

| 检查项 | 说明 |
|--------|------|
| 枚举值同步 | 新增枚举值是否同步到枚举定义源文档？ |
| Protobuf 字段同步 | 字段变更是否同步到 proto 定义和 JSON Schema？ |
| 状态机同步 | 状态变更是否同步到相关文档的状态机？ |
| 术语同步 | 新增概念是否同步到 README/CLAUDE/project_rules？ |
| 目录结构同步 | 目录变更是否同步到 README/CLAUDE 的项目结构？ |

### 数据模型一致性

| 检查项 | 说明 |
|--------|------|
| 字段类型匹配 | 同一概念在不同文档中的类型是否一致？ |
| 字段命名一致 | 同一概念在不同文档中的命名是否一致？ |
| 枚举值覆盖 | 枚举值是否完整覆盖所有文档中使用的值？ |
| Protobuf 字段编号 | 字段编号是否与已有定义冲突？ |

### 接口契约一致性

| 检查项 | 说明 |
|--------|------|
| gRPC 接口定义 | Service/RPC 方法签名是否与调用方期望一致？ |
| Go 接口定义 | 接口方法签名是否与 Protobuf 定义对应？ |
| 错误码定义 | 错误码是否与已有定义冲突？ |

## 评审报告格式

评审完成后输出结构化报告：

```markdown
## [文档名称] 评审报告（vX.Y → vX.Z）

### 🔴 阻塞项（N 个）

| 编号 | 问题 | 修复内容 |
|------|------|---------|
| B-1 | 问题描述 | 修复说明 |

### 🟡 建议项（N 个）

| 编号 | 问题 | 修复内容 |
|------|------|---------|
| S-1 | 问题描述 | 修复说明 |

### 工程哲学验证

| 原则 | 验证结果 |
|------|---------|
| Flow First | ✅/❌ 说明 |
| Deterministic Over Smart | ✅/❌ 说明 |
| Artifact Over Conversation | ✅/❌ 说明 |
| Agent Is Replaceable | ✅/❌ 说明 |
```

## 代码评审要求

1. 代码必须追溯到对应的设计文档
2. 代码必须通过自动化检查（lint/type-check/test）
3. PR 必须包含变更说明和设计文档引用
4. 代码评审同样使用 B/S 分级标记问题
